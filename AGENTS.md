# AGENTS

本文件用于约定在本仓库内工作的 AI/自动化代理行为规范。

## 目标

- 保持改动最小且可验证。
- 优先修复问题与提升可维护性。
- 不引入无关重构。

## 仓库概览

- `backend`：FastAPI 后端、业务服务、后台接口、任务轮询。
- `bot_buyer`：买家 Telegram 机器人。
- `bot_session`：客服会话 Telegram 机器人。
- `bot_shipping`：供应链 Telegram 机器人。
- `docker-compose.yml`：本地/部署的基础编排入口。

## 代码修改规则

1. 先读后改：先定位入口与调用链，再做改动。
2. 小步提交：每次改动聚焦单一问题。
3. 不做无关清理：避免“顺手重构”造成风险扩散。
4. 保持风格一致：遵循现有命名、结构与注释风格。
5. 涉及接口变更时，同步检查调用方（尤其三类 bot）。

## 严格限制

- 未经明确要求，不得修改基础设施/部署行为：
  - `docker-compose.yml`
  - Dockerfile
  - 数据库连接与密钥策略
- 未经明确要求，不得批量重命名文件或目录。
- 未经明确要求，不新增外部依赖。

## 环境变量与敏感信息

- 不将真实密钥写入仓库。
- 新增环境变量时：
  1. 在 `.env.example` 增加占位项；
  2. 在文档中说明用途与默认值；
  3. 避免破坏现有默认行为。

## 测试与验证建议

改动后至少执行与改动相关的最小验证：

- 后端改动：
  - 启动后检查 `/health`；
  - 验证对应 API 的正反向路径。
- Bot 改动：
  - 验证启动、拉取配置、核心命令流程；
  - 验证与 backend 的内部鉴权链路。
- 导出/物流改动：
  - 验证导出文件可下载且格式可打开。

## 提交说明建议

提交信息建议使用动词开头并说明意图，例如：

- `fix: handle payment refresh status race`
- `feat: add supplier-scoped shipping export`
- `refactor: split chat session service helpers`

## 文档维护

当出现以下变更时，必须同步更新文档：

- 新增服务或目录职责变化；
- 新增/删除环境变量；
- 启动方式或端口映射变化；
- 核心业务流程变化（支付、物流、会话）。
